This walkthrough takes you from a fresh account to a completed agent run via
the brenlabs API. Everything here is also doable in the dashboard at
[bren.run](https://bren.run) — the API path is shown so you can script it.

## Prerequisites

- A brenlabs account at [bren.run](https://bren.run). The MVP is password-only,
  no social login.
- `curl` and `jq` for the examples below.
- An [Anthropic Console API key](https://console.anthropic.com/settings/keys)
  if you plan to run Claude Code agents at scale — OAuth tokens work but
  expire roughly every eight hours.

## 1. Get an access token

After signing up, the dashboard issues a session JWT. Copy it from your
**Profile → API access** page or grab it from a logged-in browser session.
Export it for use below:

```bash
export BREN_JWT="eyJhbGciOi..."   # your session JWT
export BREN_ORG="org_..."         # your org id, visible in the dashboard URL
```

All API calls require both an `Authorization: Bearer <jwt>` header and an
`X-Org-Id: <org>` header to scope the request.

## 2. Register a product

A **product** is the system the agent will be tested against. Even if your
scenario doesn't actually call the product URL (e.g. for a pure docs-writing
test), brenlabs requires every run to be attached to one.

```bash
curl -sS -X POST https://api.bren.run/v1/products \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Linear",
    "slug": "linear",
    "description": "Linear issue tracker",
    "url": "https://linear.app"
  }' | jq '.id'
```

Save the returned `prod_…` id.

## 3. Create an agent

An **agent** is the configuration you're testing — which underlying tool
(`claude-code`, `codex`), which model, which skills, which MCP servers. Agents
are reusable across scenarios.

```bash
curl -sS -X POST https://api.bren.run/v1/agents \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "claude-code-default",
    "slug": "claude-code-default",
    "agent_type": "claude-code",
    "model": "claude-haiku-4-5",
    "system_prompt": "You are a focused engineer. Read the scenario carefully before acting."
  }' | jq '.id'
```

Save the returned `ag_…` id.

## 4. Write a scenario

A **scenario** carries the task description shown to the agent plus the
success criteria the judge will use. Criteria stay server-side — the agent
never sees them.

```bash
curl -sS -X POST https://api.bren.run/v1/scenarios \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "smoke-test",
    "slug": "smoke-test",
    "description_template": "Print the current date in ISO-8601 format.",
    "success_criteria": "The agent printed a valid ISO-8601 date string in its final response."
  }' | jq '.id'
```

Save the returned `scen_…` id.

## 5. Start a run

A **run** binds a product, agent, and scenario together and kicks off
execution.

```bash
curl -sS -X POST https://api.bren.run/v1/runs \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG" \
  -H "Content-Type: application/json" \
  -d '{
    "product_id": "prod_...",
    "agent_id":   "ag_...",
    "scenario_id":"scen_..."
  }' | jq '.id'
```

You'll get back a `run_…` id immediately. The run executes asynchronously.

## 6. Watch the run

Stream the live event channel:

```bash
curl -N "https://api.bren.run/v1/runs/run_.../stream" \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG"
```

Or poll for the final state:

```bash
curl -sS "https://api.bren.run/v1/runs/run_..." \
  -H "Authorization: Bearer $BREN_JWT" \
  -H "X-Org-Id: $BREN_ORG" \
  | jq '{state, judge_passed, judge_confidence, state_reason}'
```

## What just happened

<Steps>
  <Step title="Sandbox creation">
    brenlabs created a Modal sandbox from the `claude-code` base image, with
    your agent's MCP config and skills materialised onto its filesystem.
  </Step>
  <Step title="Agent execution">
    The CLI ran inside the sandbox with the scenario as stdin. Every tool
    call, thinking block, and text response streamed back to brenlabs.
  </Step>
  <Step title="Judge evaluation">
    A separate model read the full transcript and your hidden criteria,
    then produced a structured verdict.
  </Step>
  <Step title="Archive">
    Raw event log and analysis JSON were uploaded to object storage so the
    run is replayable later.
  </Step>
</Steps>
