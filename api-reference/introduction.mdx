## Authentication

Every request needs two headers:

| Header | Value |
|---|---|
| `Authorization` | `Bearer <jwt>` for user sessions, or `Bearer brn_live_…` for API tokens |
| `X-Org-Id` | The org id the request is scoped to |

User sessions come from the dashboard at [bren.run](https://bren.run).
API tokens can be created at **Settings → API tokens** for headless
automation.

Some endpoints (secrets management, org owner actions) require a user
session — API tokens get `403 tokens_cannot_*` on those. The endpoint
description in each page notes the constraint where it applies.

## Error shape

All errors follow the same JSON shape:

```json
{
  "error": "skill_not_found",
  "message": "no such skill",
  "details": { "skill_id": "skill_xxx" }
}
```

`error` is a stable, machine-readable code. `message` is human-friendly.
`details` carries structured context where useful.

## Idempotency

Mutating endpoints accept an `Idempotency-Key` header. Replays within 24
hours return the original response without re-executing.

## Cursor-based pagination

List endpoints return:

```json
{ "items": [...], "next_cursor": "..." }
```

When `next_cursor` is non-null, pass it as `?cursor=<value>` on the next
request. When it's null, you've reached the end.
